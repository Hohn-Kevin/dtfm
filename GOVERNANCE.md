# Governance

DTFM is currently maintained under a **benevolent maintainer** model appropriate for an early-stage project.

## Roles

### Maintainer

The maintainer is responsible for:

- project direction and scope;
- repository administration;
- release decisions;
- security coordination;
- reviewing and merging contributions;
- appointing additional maintainers;
- enforcing project policies and the Code of Conduct.

The current maintainer is the repository owner, `Hohn-Kevin`.

### Contributor

A contributor is anyone who participates through issues, documentation, design review, code, testing, security analysis, or other accepted project work.

Contribution does not automatically grant commit access or decision-making authority.

## Decision-making

Decisions should be based on documented requirements, technical evidence, security, maintainability, and project goals.

The project seeks constructive consensus. When consensus cannot be reached, the maintainer makes the final decision and should document significant reasoning.

## Architecture decisions

Changes with substantial or long-term architectural impact require an Architecture Decision Record under `docs/decisions/`.

An ADR should describe:

- context and problem;
- decision drivers;
- considered options;
- chosen decision;
- consequences and trade-offs;
- status.

Accepted ADRs remain part of the historical record. Superseded decisions should be marked as superseded rather than deleted.

## Maintainer appointment

Additional maintainers may be appointed based on sustained, trustworthy participation. Relevant factors include:

- quality and consistency of contributions;
- sound technical judgment;
- respectful collaboration;
- security and privacy awareness;
- willingness to perform review and maintenance work;
- alignment with documented project principles.

## Changes to governance

Governance changes must be proposed through a pull request and clearly explained. Major changes should remain open long enough for community review when an active contributor community exists.

## Project continuity

If the current maintainer can no longer maintain DTFM, reasonable efforts should be made to transfer responsibility to trusted active contributors or clearly archive the repository.
